# 역할별 온보딩 상태가 분리되지 않아 자녀 앱 진입 시 잘못된 홈 흐름으로 이동한 문제

### 문제 상황

효잇은 하나의 앱 안에서 부모 사용자와 자녀 사용자가 각각 다른 화면 흐름을 가진다. 로그인 이후 사용자가 역할을 선택하면, 선택한 역할에 따라 부모용 온보딩 또는 자녀용 온보딩으로 이동하고, 온보딩이 완료된 이후에는 각 역할에 맞는 홈 화면으로 진입해야 했다.

기대했던 흐름은 다음과 같았다.

```
자녀 선택
→ 자녀 온보딩
→ 자녀 온보딩 완료
→ 자녀 전용 홈 화면
```

하지만 실제로는 자녀 역할을 선택했는데도 자녀 전용 홈 화면으로 자연스럽게 이어지지 않고, 부모 앱의 홈 흐름처럼 보이는 문제가 발생했다. 특히 `ChildHomePage`를 단순하게 `Child Home` 텍스트만 렌더링하도록 만들어도 화면 흐름이 예상과 다르게 동작해, 처음에는 라우팅 구조 문제로 판단했다.

당시 자녀 홈 라우트는 다음처럼 정상적으로 연결되어 있었다.

```tsx
import ChildHomePage from "@/src/child/widgets/home-page";

export default function ChildHomeRoute() {
  return <ChildHomePage />;
}
```

그리고 자녀 홈 화면도 부모 홈을 참조하지 않고 있었다.

```tsx
import { StyleSheet, Text, View } from "react-native";

export default function ChildHomePage() {
  return (
    <View style={styles.container}>
      <Text style={styles.text}>Child Home</Text>
    </View>
  );
}

const styles = StyleSheet.create({
  container: {
    flex: 1,
    alignItems: "center",
    justifyContent: "center",
    backgroundColor: "#FFFFFF",
  },
  text: {
    fontSize: 20,
    fontWeight: "600",
    color: "#000000",
  },
});
```

그럼에도 역할별 홈 흐름이 명확하게 분리되지 않는 문제가 계속 발생했다.

---

### 처음 의심한 원인: Expo Router route group 구조

처음에는 Expo Router의 route group 구조를 의심했다.

당시 라우팅 구조는 다음과 같았다.

```
app/
  (parent)/
    index.tsx
    (tabs)/
      index.tsx

  (child)/
    index.tsx
    (tabs)/
      index.tsx
```

`(parent)`, `(child)`, `(tabs)`는 Expo Router에서 URL에 직접 포함되지 않는 route group이다. 그래서 처음에는 부모와 자녀 모두 `(tabs)/index.tsx`를 가지고 있어 홈 경로가 충돌하는 것처럼 보인다고 판단했다.

이를 해결하기 위해 `app/(child)`를 `app/child`로 바꾸거나, `(tabs)/index.tsx`를 `(tabs)/home.tsx`로 변경하는 방향도 시도했다.

하지만 이 과정에서 Expo Router typed routes가 기존 route group 기반 경로를 기준으로 타입을 생성하고 있어 다음과 같은 타입 에러가 발생했다.

```
Type '"/child/home"' is not assignable to type ...
Did you mean '"/(child)/home"'?
```

또는 다음처럼 `(tabs)` 경로까지 포함해야 하는 것처럼 보이는 에러도 발생했다.

```
Type '"/(child)/(tabs)/home"' is not assignable to type ...
Did you mean '"/(child)/(tabs)"'?
```

이 과정을 통해 단순히 폴더명을 바꾸는 방식은 오히려 라우팅 타입과 실제 구조를 더 혼란스럽게 만들 수 있다고 판단했다. 그래서 라우트 구조를 무리하게 변경하기보다, 기존 route group 구조는 유지하고 실제 분기 기준이 되는 상태 값을 다시 확인했다.

### 실제 원인: 부모와 자녀가 하나의 `hasOnboarded` 상태를 공유하고 있었음

문제의 실제 원인은 라우팅 파일이 아니라 인증 store 구조에 있었다.

기존에는 부모와 자녀의 온보딩 완료 여부를 하나의 boolean 값으로 관리하고 있었다.

```tsx
interface AuthState {
  isSignedIn: boolean;
  role: UserRole | null;
  hasOnboarded: boolean;

  setSignedIn: (value: boolean) => void;
  setRole: (role: UserRole | null) => void;
  setOnboarded: (value: boolean) => void;
  resetAuth: () => void;
}
```

이 구조에서는 부모 온보딩을 완료해도 `hasOnboarded`가 `true`가 되고, 자녀 온보딩을 완료해도 동일하게 `hasOnboarded`가 `true`가 된다.

즉 앱 입장에서는 다음 두 상태를 구분할 수 없었다.

```
부모 온보딩 완료
자녀 온보딩 완료
```

그 결과 부모 앱과 자녀 앱이 서로 다른 온보딩 플로우를 가져야 하는데도, 둘 다 같은 `hasOnboarded` 값을 기준으로 분기하게 되었다.

기존 부모 진입부는 다음과 같았다.

```tsx
import { useAuthStore } from "@hyoit/auth";
import { Redirect } from "expo-router";

export default function ParentIndex() {
  const hasOnboarded = useAuthStore((s) => s.hasOnboarded);

  if (!hasOnboarded) {
    return <Redirect href="/(parent)/onboarding/child-info" />;
  }

  return <Redirect href="/(parent)/(tabs)" />;
}
```

자녀 진입부도 같은 값을 보고 있었다.

```tsx
import { useAuthStore } from "@hyoit/auth";
import { Redirect } from "expo-router";

export default function ChildIndex() {
  const hasOnboarded = useAuthStore((s) => s.hasOnboarded);

  if (!hasOnboarded) {
    return <Redirect href="/(child)/onboarding/parent-info" />;
  }

  return <Redirect href="/(child)/(tabs)" />;
}
```

이 구조에서는 부모와 자녀의 온보딩 완료 여부를 독립적으로 판단할 수 없었다.

### 추가로 발견한 문제: `useAuthStore`가 중복으로 존재함

상태 구조를 확인하는 과정에서 `useAuthStore`가 두 곳에 존재하는 것도 발견했다.

```
grep -R "export const useAuthStore" apps packages
```

검색 결과는 다음과 같았다.

```
apps/mobile/src/parent/entities/auth/model/authStore.ts
packages/auth/model/authStore.ts
```

하나는 앱 내부 parent 도메인에 존재했고, 다른 하나는 모노레포 공용 패키지인 `packages/auth`에 존재했다.

```
apps/mobile/src/parent/entities/auth/model/authStore.ts
packages/auth/model/authStore.ts
```

대부분의 화면에서는 다음처럼 공용 패키지의 store를 사용하고 있었다.

```tsx
import { useAuthStore } from "@hyoit/auth";
```

따라서 `packages/auth/model/authStore.ts`가 실제 기준 store였고, `apps/mobile/src/parent/entities/auth/model/authStore.ts`는 이전 구조에서 남아 있는 중복 store였다.

인증 상태는 앱 전역에서 하나의 기준으로 관리되어야 하므로, parent 내부에 남아 있던 중복 auth store는 제거 대상으로 판단했다.

### 해결 방법 1: 온보딩 상태를 역할별로 분리

먼저 공용 auth store에서 `hasOnboarded`를 제거하고, 부모와 자녀의 온보딩 완료 상태를 별도로 분리했다.

수정 전 구조는 다음과 같았다.

```tsx
interface AuthState {
  isSignedIn: boolean;
  role: UserRole | null;
  hasOnboarded: boolean;

  setSignedIn: (value: boolean) => void;
  setRole: (role: UserRole | null) => void;
  setOnboarded: (value: boolean) => void;
  resetAuth: () => void;
}
```

수정 후 구조는 다음과 같다.

```tsx
import { create } from "zustand";
import type { UserRole } from "@hyoit/types";

interface AuthState {
  isSignedIn: boolean;
  role: UserRole | null;
  hasParentOnboarded: boolean;
  hasChildOnboarded: boolean;

  setSignedIn: (value: boolean) => void;
  setRole: (role: UserRole | null) => void;
  setParentOnboarded: (value: boolean) => void;
  setChildOnboarded: (value: boolean) => void;
  resetAuth: () => void;
}

export const useAuthStore = create<AuthState>((set) => ({
  isSignedIn: false,
  role: null,
  hasParentOnboarded: false,
  hasChildOnboarded: false,

  setSignedIn: (value) => set({ isSignedIn: value }),
  setRole: (role) => set({ role }),
  setParentOnboarded: (value) => set({ hasParentOnboarded: value }),
  setChildOnboarded: (value) => set({ hasChildOnboarded: value }),

  resetAuth: () =>
    set({
      isSignedIn: false,
      role: null,
      hasParentOnboarded: false,
      hasChildOnboarded: false,
    }),
}));
```

이렇게 수정하면서 부모 온보딩 완료 여부와 자녀 온보딩 완료 여부를 독립적으로 관리할 수 있게 되었다.

---

### 해결 방법 2: 부모와 자녀의 진입 분기 기준 분리

부모 앱 진입부에서는 `hasParentOnboarded`만 확인하도록 수정했다.

```tsx
import { useAuthStore } from "@hyoit/auth";
import { Redirect } from "expo-router";

export default function ParentIndex() {
  const hasParentOnboarded = useAuthStore((s) => s.hasParentOnboarded);

  if (!hasParentOnboarded) {
    return <Redirect href="/(parent)/onboarding/child-info" />;
  }

  return <Redirect href="/(parent)/(tabs)" />;
}
```

자녀 앱 진입부에서는 `hasChildOnboarded`만 확인하도록 수정했다.

```tsx
import { useAuthStore } from "@hyoit/auth";
import { Redirect } from "expo-router";

export default function ChildIndex() {
  const hasChildOnboarded = useAuthStore((s) => s.hasChildOnboarded);

  if (!hasChildOnboarded) {
    return <Redirect href="/(child)/onboarding/parent-info" />;
  }

  return <Redirect href="/(child)/(tabs)" />;
}
```

처음에는 `role !== "parent"` 또는 `role !== "child"`일 때 루트로 돌려보내는 guard도 추가했지만, 디버깅 중에는 role 저장 시점과 redirect 흐름이 맞물리면서 오히려 문제 파악을 어렵게 만들 수 있었다. 그래서 우선은 온보딩 분기 기준을 명확히 분리하는 데 집중했다.

---

### 해결 방법 3: 온보딩 성공 화면에서 역할별 setter 호출

기존 부모 온보딩 성공 화면에서는 공용 setter인 `setOnboarded(true)`를 호출하고 있었다.

```tsx
const setOnboarded = useAuthStore((state) => state.setOnboarded);

const start = () => {
  setOnboarded(true);
  router.replace("/(parent)");
};
```

이를 부모 전용 setter로 변경했다.

```tsx
const setParentOnboarded = useAuthStore((state) => state.setParentOnboarded);

const start = () => {
  setParentOnboarded(true);
  router.replace("/(parent)");
};
```

자녀 온보딩 성공 화면에서도 동일하게 `setChildOnboarded(true)`를 호출하도록 수정했다.

```tsx
const setChildOnboarded = useAuthStore((state) => state.setChildOnboarded);

const start = () => {
  setChildOnboarded(true);
  router.replace("/(child)");
};
```

이렇게 수정하면서 부모 온보딩 완료는 부모 상태에만, 자녀 온보딩 완료는 자녀 상태에만 반영되도록 만들었다.

### 검증 과정

수정 후 기존 공용 온보딩 상태가 남아 있는지 검색했다.

```
grep -R "hasOnboarded\|setOnboarded" apps packages
```

수정 전에는 다음처럼 부모 온보딩 성공 화면에서 기존 상태가 남아 있었다.

```
apps/mobile/src/parent/widgets/onboarding-page/ui/SuccessScreen.tsx:
  const setOnboarded = useAuthStore((state) => state.setOnboarded);

apps/mobile/src/parent/widgets/onboarding-page/ui/SuccessScreen.tsx:
  setOnboarded(true);
```

해당 부분을 모두 `setParentOnboarded`로 변경한 뒤 다시 검색했을 때 결과가 나오지 않는 것을 확인했다.

또한 중복 store 사용 여부를 확인하기 위해 다음 명령어도 사용했다.

```
grep -R "export const useAuthStore" apps packages
```

이를 통해 auth store의 위치가 중복되어 있음을 확인했고, 실제 앱에서 사용하는 `@hyoit/auth` 기준으로 상태 관리를 통일하는 방향으로 정리했다.

### 결과

수정 후 부모와 자녀의 온보딩 상태가 독립적으로 관리되었다.

최종 흐름은 다음과 같이 분리되었다.

```
부모 선택
→ 부모 온보딩
→ setParentOnboarded(true)
→ 부모 홈 진입
```

```
자녀 선택
→ 자녀 온보딩
→ setChildOnboarded(true)
→ 자녀 홈 진입
```

이전에는 `hasOnboarded` 하나가 부모와 자녀에게 공통으로 적용되면서 역할별 흐름이 섞였지만, 상태를 역할별로 나누면서 각 앱의 진입 흐름이 명확해졌다.

---

### 배운 점

이번 문제는 처음에는 Expo Router의 route group이나 `(tabs)/index.tsx` 구조 문제처럼 보였다. 하지만 실제 원인은 라우팅 자체가 아니라, 라우팅 분기의 기준이 되는 상태 값이 역할별로 분리되어 있지 않은 데 있었다.

라우팅 문제를 해결할 때는 경로 구조만 볼 것이 아니라, 해당 경로로 이동하게 만드는 상태 값이 올바르게 설계되어 있는지 함께 확인해야 한다.

특히 효잇처럼 하나의 앱 안에서 부모와 자녀라는 서로 다른 사용자 역할을 다루는 경우, 단순히 `hasOnboarded` 같은 공용 boolean 하나로 상태를 관리하면 역할별 흐름이 쉽게 섞일 수 있다. 역할별로 다른 화면 흐름을 가진다면 상태 역시 역할별로 분리되어야 한다.

또한 모노레포 구조에서는 공용 상태 store와 앱 내부 상태 store가 중복으로 존재하지 않도록 책임 위치를 명확히 해야 한다. 인증 상태처럼 앱 전체에서 공유되는 상태는 `packages/auth`와 같은 공용 패키지에서 일관되게 관리하는 것이 더 안전하다.
