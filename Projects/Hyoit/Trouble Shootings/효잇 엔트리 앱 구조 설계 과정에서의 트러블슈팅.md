# 효잇 엔트리 앱 구조 설계 과정에서의 트러블슈팅

## Expo 모노레포 환경에서 Entry / Parent / Child 구조를 어떻게 정의할 것인가

## 1. 문제 배경

효잇 프로젝트는 부모와 자녀가 각각 다른 흐름으로 서비스를 이용하는 구조를 갖고 있다.

초기 기획 단계에서는 앱 실행 직후 `entry` 영역에서 로그인 및 역할 선택을 수행한 뒤, 사용자의 역할이 부모인지 자녀인지에 따라 각각 다른 앱 흐름으로 진입시키는 구조를 구상하였다.

이 과정에서 프로젝트는 다음과 같은 디렉토리 구조로 설계되었다.

```
apps/
  entry/
  parent/
  child/
packages/
  auth/
  storage/
  types/
  ui/
```

표면적으로 이 구조는 역할별 앱을 분리한 것처럼 보였고, 모노레포를 활용해 공통 로직과 UI를 공유하려는 의도를 담고 있었다. 그러나 실제 구현을 진행하면서 다음과 같은 근본적인 질문이 발생하였다.

- `entry`, `parent`, `child`를 정말 서로 다른 “앱”으로 취급해야 하는가?
- Expo 환경에서 앱 간 전환은 어떤 방식으로 이루어져야 하는가?
- 최종 배포 대상이 하나의 모바일 앱이라면, 현재 구조는 적절한가?
- 모노레포를 사용한다는 이유만으로 실행 단위를 여러 개로 나누는 것이 옳은가?

이 문제는 단순한 폴더 구조의 선택이 아니라, **프로젝트의 실행 단위와 도메인 경계를 어떻게 정의할 것인가**에 대한 설계 문제로 이어졌다.

---

## 2. 초기 설계와 그 의도

초기에는 `entry`, `parent`, `child`를 각각 `apps` 아래에 두었다.

이 구조를 선택한 이유는 다음과 같았다.

첫째, 부모와 자녀의 화면 흐름이 명확히 분리되어 있었기 때문이다.

둘째, 엔트리 단계에서 사용자 역할을 분기한 뒤 각 흐름으로 자연스럽게 이동시키고 싶었기 때문이다.

셋째, 공통 인증 로직과 UI는 `packages`로 추출해 모노레포의 장점을 살리고자 했기 때문이다.

이 설계는 겉보기에 다음과 같은 흐름을 상정하고 있었다.

```
entry 앱 실행
→ 로그인
→ 역할 선택
→ parent 앱 또는 child 앱으로 이동
```

즉, `entry`가 하나의 게이트웨이 앱처럼 동작하고, 이후 `parent` 또는 `child` 앱으로 넘겨주는 모델이었다.

초기에는 이 구조가 논리적으로 잘 맞는 것처럼 보였다.

하지만 실제로 Expo 환경에서 이를 어떻게 테스트하고 구현할 수 있는지 검토하는 과정에서 문제가 드러나기 시작했다.

---

## 3. 문제 정의

핵심 문제는 다음과 같이 정리할 수 있었다.

### 3.1 실행 단위와 서비스 단위가 혼동되고 있었다

효잇은 실제로는 하나의 모바일 서비스로 배포될 예정이었다.

즉 App Store 혹은 Play Store에 올라가는 것은 “효잇”이라는 단일 앱이다.

그 안에서 부모 사용자와 자녀 사용자가 서로 다른 흐름을 갖는 구조일 뿐, 배포 결과물이 서로 다른 세 개의 앱이 되는 것은 아니었다.

그럼에도 불구하고 `apps/entry`, `apps/parent`, `apps/child` 구조는 자연스럽게 다음과 같은 오해를 낳았다.

- 엔트리 앱이 별도 앱이다
- 부모 앱과 자녀 앱도 별도 앱이다
- 사용자는 앱 내부 분기가 아니라 앱 간 이동을 하게 된다

즉, **도메인 흐름의 분리**를 **실행 애플리케이션의 분리**로 잘못 해석하고 있었던 것이다.

---

### 3.2 Expo 모노레포 지원과 앱 간 실행 문제를 같은 문제로 보고 있었다

Expo 공식 문서를 통해 모노레포 구성이 가능하다는 점은 확인할 수 있었다.

실제로 Expo는 pnpm workspace 기반 monorepo를 지원하며, `apps/*`, `packages/*` 구조 역시 일반적으로 사용되는 패턴이다.

그러나 여기서 확인된 것은 어디까지나 다음과 같은 사실이었다.

- 여러 Expo 프로젝트를 하나의 저장소에서 관리할 수 있다
- 공통 패키지를 workspace로 공유할 수 있다
- 모노레포 환경에서도 Expo 앱을 실행할 수 있다

반면, 다음의 문제는 전혀 다른 층위에 있었다.

- `entry` 앱이 `parent` 앱이나 `child` 앱을 “실제로 실행”할 수 있는가?
- Expo Go에서 앱 간 전환이 가능한가?
- 딥링크나 scheme 없이 별도 앱을 오갈 수 있는가?

즉, **“Expo에서 monorepo가 된다”는 사실과 “앱 간 실행이 자연스럽게 가능하다”는 결론은 동일하지 않았다.**

---

### 3.3 현재 단계에서 필요하지 않은 복잡성을 먼저 끌어오고 있었다

당시 `navigateToTarget("parent")`, `navigateToTarget("child")`와 같은 추상화가 존재했고, 이 함수는 부모 앱 또는 자녀 앱으로 이동한다는 의미를 담고 있었다.

하지만 실제 구현은 다음과 같았다.

```
case"parent":
console.log("Go to parent app");
return;

case"child":
console.log("Go to child app");
return;
```

이는 아직 실제 앱 간 이동 전략이 정해지지 않았다는 것을 보여준다.

그럼에도 불구하고 구조상으로는 마치 별도 앱 간 전환을 당장 구현해야 하는 것처럼 설계가 흘러가고 있었다.

즉, 아직 확정되지 않은 통합 연결 문제를 너무 이른 시점에 구조의 중심에 두고 있었던 것이다.

---

## 4. 검토 과정

이 문제를 해결하기 위해 다음과 같은 질문을 기준으로 구조를 다시 검토하였다.

### 4.1 실제 배포 단위는 몇 개인가?

가장 먼저 확인한 것은 “효잇은 최종적으로 몇 개의 앱으로 배포되는가”였다.

결론은 명확했다.

- 배포 앱은 하나다
- 서비스 이름도 하나다
- 사용자 입장에서도 하나의 앱 안에서 부모/자녀 흐름이 분기되는 구조가 자연스럽다

이 사실이 확인되자, `entry`, `parent`, `child`를 독립 실행 앱처럼 바라보는 관점은 설득력을 잃기 시작했다.

---

### 4.2 역할 분기는 앱 간 이동인가, 앱 내부 라우팅인가?

다음으로 검토한 것은 역할 분기의 본질이었다.

초기에는 역할 선택 이후 “부모 앱으로 이동”, “자녀 앱으로 이동”이라는 표현을 사용했지만, 실질적으로 필요한 것은 다음과 같은 흐름이었다.

```
앱 실행
→ 로그인 여부 확인
→ 역할 여부 확인
→ 부모 화면 흐름 또는 자녀 화면 흐름 진입
```

이 과정은 엄밀히 말해 **앱 간 전환**이 아니라 **앱 내부의 초기 라우팅 분기**에 가깝다.

즉 문제의 본질은 다음과 같았다.

- 독립 앱 실행이 필요한가? → 아니다
- 내부 route group 분기가 필요한가? → 그렇다

이 지점에서 구조를 “여러 앱” 중심이 아니라 “하나의 앱 내부 흐름” 중심으로 재해석할 수 있게 되었다.

---

### 4.3 모노레포의 목적은 정말 여러 앱 운영인가?

초기에는 모노레포를 사용한다는 사실 자체가 `apps/entry`, `apps/parent`, `apps/child` 같은 구조를 정당화하는 근거처럼 작용하고 있었다.

그러나 다시 생각해보면, 모노레포의 본질은 여러 앱을 반드시 운영하는 데 있지 않다.

모노레포의 핵심 가치는 다음과 같다.

- 실행 앱과 공유 코드를 명확히 분리할 수 있다
- `auth`, `ui`, `types`, `storage` 같은 공통 관심사를 독립 패키지로 관리할 수 있다
- 장기적으로 admin, web, landing, docs 등의 확장에 유리하다
- 도메인 경계를 코드 구조에 반영할 수 있다

즉, **모바일 앱이 하나만 있어도 모노레포는 충분히 유효하다.**

이 판단을 통해 “앱이 하나면 모노레포 의미가 없다”는 인식이 잘못되었음을 확인할 수 있었다.

---

## 5. 최종 판단

위 검토를 통해 다음과 같은 결론에 도달하였다.

### 5.1 `entry`, `parent`, `child`는 앱이 아니라 흐름이다

이 세 가지는 독립된 배포 애플리케이션이 아니다.

하나의 효잇 앱 내부에서 역할에 따라 나뉘는 사용자 흐름일 뿐이다.

따라서 `apps/entry`, `apps/parent`, `apps/child`라는 구조는 실행 단위를 과도하게 분리한 설계로 판단하였다.

---

### 5.2 실제 실행 앱은 하나만 두는 것이 맞다

배포 구조와 실행 구조를 일치시키기 위해, 실제 애플리케이션은 하나만 두고 내부에서 역할별 route group을 나누는 방식이 더 적절하다고 결론 내렸다.

즉 권장 구조는 다음과 같다.

```
apps/
  mobile/
packages/
  auth/
  storage/
  types/
  ui/
```

그리고 `mobile` 앱 내부에서는 다음과 같이 분기한다.

```
app/
  (entry)/
    login.tsx
    choose.tsx
  (parent)/
    ...
  (child)/
    ...
```

이렇게 하면 실행 단위는 하나로 유지하면서도 역할별 흐름은 충분히 명확하게 분리할 수 있다.

---

### 5.3 `navigateToTarget`은 앱 간 실행이 아니라 내부 라우팅으로 재해석한다

기존 `navigateToTarget`의 추상화 자체는 유효했다.

문제는 그것이 “다른 앱을 실행하는 함수”처럼 이해되고 있었다는 점이다.

이를 다음과 같이 재정의할 수 있다.

- `login` → entry 로그인 화면으로 이동
- `choose` → entry 역할 선택 화면으로 이동
- `parent` → parent route group의 시작 화면으로 이동
- `child` → child route group의 시작 화면으로 이동

즉 이 함수는 앞으로도 유지할 수 있지만, 역할은 **외부 앱 전환**이 아니라 **내부 route 분기 유틸**로 바뀌어야 한다.

---

## 6. 최종 구조 제안

최종적으로는 다음과 같은 구조가 가장 적절하다고 판단하였다.

```
apps/
  mobile/
    app/
      _layout.tsx
      index.tsx
      (entry)/
        login.tsx
        choose.tsx
      (parent)/
        ...
      (child)/
        ...
packages/
  auth/
    model/
    lib/
    api/
  storage/
  types/
  ui/
```

이 구조의 장점은 다음과 같다.

첫째, 실제 배포 형태와 코드 구조가 일치한다.

둘째, 역할별 흐름이 route group 수준에서 명확히 구분된다.

셋째, `auth`, `types`, `ui`, `storage` 같은 공통 기반은 여전히 모노레포로 관리할 수 있다.

넷째, 향후 admin 웹이나 별도의 백오피스 프로젝트가 추가되더라도 자연스럽게 확장할 수 있다.

---

## 7. 이번 트러블슈팅을 통해 얻은 교훈

이번 문제는 단순히 “Expo에서 monorepo가 되느냐”를 확인하는 수준의 이슈가 아니었다.

실제로는 다음 두 가지를 명확히 구분하지 못해서 발생한 설계 혼선이었다.

### 7.1 서비스 흐름 분리와 실행 앱 분리는 다른 문제다

부모와 자녀의 경험이 다르다는 이유만으로 곧바로 별도 앱 구조를 떠올리면, 실제 배포 단위와 실행 단위가 어긋날 수 있다.

도메인 분리는 route, feature, package 수준에서도 충분히 달성할 수 있다.

---

### 7.2 모노레포는 “앱 개수”보다 “경계 관리”를 위한 도구다

모노레포의 본질은 여러 앱을 억지로 만드는 데 있지 않다.

오히려 하나의 앱이더라도, 공통 기반과 실행 코드를 분리하고, 도메인 경계를 유지하며, 장기 확장을 준비하는 데 더 큰 의미가 있다.

---

## 8. 결론

효잇 프로젝트에서 발생한 구조 설계 문제는 “Expo에서 monorepo가 가능한가”가 아니라, **하나의 서비스 안에서 부모/자녀 흐름을 어떤 단위로 나눌 것인가**에 대한 문제였다.

초기에는 `entry`, `parent`, `child`를 각각 독립 앱처럼 두는 구조를 고려했으나, 최종적으로는 다음과 같은 판단에 도달하였다.

- 효잇은 하나의 앱으로 배포된다
- 따라서 실행 단위도 하나의 모바일 앱으로 유지하는 것이 자연스럽다
- 부모/자녀/엔트리의 차이는 앱 간 차이가 아니라 앱 내부 흐름의 차이다
- 모노레포는 여전히 유효하며, 실행 앱과 공유 패키지를 분리하는 데 의미가 있다

결과적으로 이 트러블슈팅은 폴더 구조를 정리하는 문제를 넘어,

**도메인 분리와 실행 단위를 어떻게 올바르게 대응시킬 것인가**에 대한 기준을 세우는 과정이었다.
