컴포넌트 책임 분리 기준
===

## 1. 이 컴포넌트는 “무엇을 보여주나”, “어떻게 바꾸나” 중 뭐를 맡는가

이걸 먼저 나눈다.

### 보여주기 담당

- 프로젝트 제목
- 이미지
- 설명
- 썸네일

이런 **내용 렌더링**만 하면 됨.

예:

- `ProjectFrameText`
- `ProjectFrameVisual`
- `ProjectFrameDetail`

이런 애들은 보통 `project` 하나만 받는 게 맞다.

---

### 바꾸기 담당

- current / next를 겹칠지
- opacity로 페이드할지
- 언제 전환 시작할지
- 언제 완료 처리할지

이건 **전환 제어**다.

예:

- `ProjectFrame`
- `ProjectsSection`

---

## 2. 이 컴포넌트가 current / next를 알아야 하는가

이 기준이 제일 중요하다.

### 알아야 하는 경우

- 두 레이어를 동시에 렌더해야 함
- 전환 타이밍을 제어해야 함
- 애니메이션을 실행해야 함

이런 경우만 `currentProject`, `nextProject`, `isTransitioning`을 알아도 된다.

보통:

- `ProjectFrame`

---

### 알 필요 없는 경우

- 프로젝트 하나 받아서 제목/이미지/설명만 그림
- 전환 중인지 몰라도 화면을 그릴 수 있음

이런 경우는 `project` 하나만 받게 한다.

보통:

- `ProjectFrameText`
- `ProjectFrameVisual`
- `ProjectFrameDetail`

---

## 3. 전환 방식이 바뀌었을 때 이 컴포넌트를 수정해야 하는가

이걸 물어보면 기준이 선다.

### 수정 안 돼야 정상인 컴포넌트

전환이

- 페이드든
- 슬라이드든
- 즉시 교체든

상관없이 그대로 써야 한다.

이런 컴포넌트는 **순수 렌더링 컴포넌트**다.

즉:

```tsx
<ProjectFrameTextproject={project}/>
```

이런 형태가 맞다.

---

### 수정될 수밖에 없는 컴포넌트

전환 방식 자체를 관리하니까

애니메이션 구조가 바뀌면 같이 수정된다.

이건 컨테이너 역할이다.

예:

- `ProjectFrame`
- `ProjectsSection`

---

## 4. 이 컴포넌트 안에 “조건부 전환 렌더”가 들어가고 있는가

예를 들어 이런 게 있으면 전환 책임이 들어온 거다.

```tsx
{isTransitioning&&nextProject&& (...)}
```

또는 이런 props가 있으면:

```
currentProject
nextProject
isTransitioning
```

이건 보통 **전환 담당 컴포넌트**에만 있어야 한다.

Text / Visual / Detail 같은 말단 렌더 컴포넌트에 이런 게 보이면

책임이 아래로 내려온 신호다.

---

## 5. ref, GSAP, cleanup이 이 컴포넌트에 꼭 필요한가

### 필요한 경우

- 실제로 애니메이션을 실행하는 곳
- 레이어를 겹쳐서 제어하는 곳

### 불필요한 경우

- 텍스트만 그리는 컴포넌트
- 이미지 하나 보여주는 컴포넌트
- 설명만 보여주는 컴포넌트

말단 컴포넌트에 `useLayoutEffect`, `gsap`, `ref`가 계속 붙기 시작하면

대체로 책임이 과하게 들어간 거다.

---

# 역할별 기준 정리

## `ProjectsSection`

**상태 결정 담당**

해야 하는 것:

- scroll → index 계산
- 전환 시작 조건 판단
- `displayIndex`, `incomingIndex`, `isTransitioning` 관리
- 전환 완료 후 state 확정

하지 말아야 하는 것:

- 텍스트/이미지 UI 직접 렌더링 세부 처리
- 하위 요소별 애니메이션 직접 관리

---

## `ProjectFrame`

**전환 무대 담당**

해야 하는 것:

- current / next 레이어 렌더
- 레이어 겹침 관리
- 배경/opacity 같은 전환 실행
- 완료 콜백 한 곳에서 처리

하지 말아야 하는 것:

- 프로젝트 개별 텍스트 내용 조립
- detail 문장 자체 생성

---

## `ProjectCard`

**프로젝트 한 장 묶음 담당**

해야 하는 것:

- `project` 하나를 받아
    - Text
    - Visual
    - Detail
        
        묶어서 렌더
        

하지 말아야 하는 것:

- current / next 판단
- isTransitioning 판단
- 애니메이션 실행

---

## `Text / Visual / Detail`

**순수 렌더 담당**

해야 하는 것:

- 받은 `project`를 화면에 표시

하지 말아야 하는 것:

- current / next 판단
- 전환 중인지 판단
- GSAP 실행
- 완료 콜백 처리

---

# 빠르게 판단하는 체크리스트

컴포넌트 만들 때 아래 질문으로 판단하면 된다.

## 이 컴포넌트가 두 개의 프로젝트를 동시에 알아야 하나?

- 예 → 상위 전환 컴포넌트일 가능성 큼
- 아니오 → 하위 렌더 컴포넌트로 두는 게 맞음

## 이 컴포넌트가 애니메이션 방식까지 알아야 하나?

- 예 → 컨테이너
- 아니오 → 프리젠테이셔널 컴포넌트

## 이 컴포넌트는 `project` 하나만으로도 렌더 가능한가?

- 예 → 말단으로 내려도 됨
- 아니오 → 상위 책임인지 다시 점검

## 이 컴포넌트가 전환 방식이 바뀔 때도 그대로 재사용돼야 하나?

- 예 → 전환 로직 넣으면 안 됨
- 아니오 → 전환 담당일 수 있음

# 지금 프로젝트에 적용하면

## 상위만 알아야 하는 값

```
displayIndex
incomingIndex
isTransitioning
currentProject
nextProject
```

## 말단은 몰라도 되는 값

```
nextProject
isTransitioning
onTransitionComplete
gsap ref
```

## 말단이 받으면 좋은 값

```
project
```
