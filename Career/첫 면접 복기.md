# 피코소프트 면접 복기

> 면접 직후 기억나는 내용을 바탕으로 정리한 기록  
> 목적: 결과 예측보다는 실제 질문, 답변, 면접관 반응, 강점과 보완점을 객관적으로 남기기

---

## 1. 면접 전체 요약

이번 면접은 예상했던 것보다 전공 및 웹 개발 기본 개념 질문의 비중이 높았다.

단순히 자기소개나 지원 동기, 프로젝트 설명만 듣는 방식이 아니라 실제 포트폴리오 웹사이트를 열고 개발자 도구까지 확인하면서 HTML 구조, 클래스 네이밍, 시맨틱 태그, 프로젝트 구현 방식 등을 깊게 검증했다.

또한 한 프로젝트만 보는 것이 아니라 개인 포트폴리오, 효잇, NOVA 등 여러 프로젝트를 계속 열어보며 다음 내용을 확인했다.

- 실제로 본인이 직접 개발했는지
- 디자인도 직접 했는지
- 학교에서 배운 내용인지 독학한 내용인지
- 기술을 왜 사용했는지 설명할 수 있는지
- 프로젝트 결과물과 기본 개념 이해도가 일치하는지
- 새로운 기술을 스스로 배우는 사람인지

전공 질문 중 일부는 답변하지 못했고, 일부는 정확한 개념보다는 알고 있는 범위에서 두루뭉술하게 답했다. 반면 프로젝트 경험, 디자인 기여도, 독학 과정, AI 활용 방식, 인성 관련 질문은 비교적 잘 연결해서 답변했다.

전체적으로 보면 기술 기본기의 약점은 드러났지만, 직접 만든 결과물의 완성도와 디자인 감각, 독학 능력, 성장 가능성도 함께 보여준 면접이었다.

---

## 2. 실제 면접 진행 방식

### 2.1 준비된 질문지만 묻는 면접이 아니었음

면접관은 준비된 전공 질문을 묻는 것에 그치지 않고 실제 웹 포트폴리오를 직접 확인했다.

특히 개발자 도구를 열어 다음과 같은 부분을 확인했다.

- HTML 태그 구조
- 클래스 네이밍
- `__`, `--` 사용 방식
- 시맨틱 태그 적용 여부
- 프로젝트 상세 모달
- 프로젝트별 디자인 기여도
- 실제 구현 세부 사항

질문은 한두 개로 끝나지 않고 여러 프로젝트를 계속 깊게 파고드는 방식이었다.

### 2.2 프로젝트 진위와 기여도 검증

면접관은 효잇과 NOVA 등의 화면을 보면서 다음과 같이 질문했다.

- “이 디자인도 직접 하신 거예요?”
- “이것도 본인이 직접 만든 건가요?”
- “학교에서 이런 걸 가르쳐주나요?”
- “독학으로 공부한 건가요?”
- “시맨틱 태그는 어떻게 알게 됐나요?”

이 질문들은 단순히 기술 지식을 확인하려는 목적뿐 아니라, 결과물의 출처와 실제 기여도를 확인하려는 성격이 강했다.

## 3. 면접관이 긍정적으로 반응한 부분

## 3.1 프로젝트 디자인

효잇과 NOVA 프로젝트를 보면서 면접관이 직접 디자인했는지를 물었고, 결과물에 대해 놀라거나 감탄하는 반응을 보였다.

특히 개인 포트폴리오의 프로젝트 상세 모달을 직접 열어서 화면을 자세히 확인했다.

이 부분은 비교적 명확한 긍정 신호로 볼 수 있다.

면접관은 다음 내용을 높게 봤을 가능성이 있다.

- 개발뿐 아니라 디자인까지 직접 할 수 있다는 점
- 화면의 완성도가 일반적인 학생 프로젝트보다 높다는 점
- 프로젝트를 단순 구현이 아니라 사용자 경험과 시각적 흐름까지 고려해 만들었다는 점
- 포트폴리오가 형식적인 결과물이 아니라 실제로 많은 고민을 거쳐 제작되었다는 점

## 3.2 독학 능력

본인은 IoT 학과이지만 학교 전공 수업에서 프론트엔드나 웹 디자인을 체계적으로 배우지 않았다.

HTML, CSS, JavaScript, React, TypeScript, 인터랙션, 디자인, 시맨틱 마크업 등은 사실상 대부분 독학으로 공부했다.

면접관은 자신도 대학에서 시맨틱 태그 등을 깊게 배우지 않았다고 말하면서, 포트폴리오에서는 실제로 시맨틱 태그를 사용한 것이 보인다고 했다.

이는 면접관이 다음과 같은 점을 확인한 것으로 해석할 수 있다.

- 학교 수업 결과물인지
- 본인이 스스로 찾아 배운 것인지
- 교육 과정 밖의 기술을 얼마나 확장했는지
- 새로운 기술을 혼자 학습할 수 있는 사람인지

결과물의 수준이 학교 교육만으로 만들어진 것으로 보이지 않았기 때문에 독학 여부를 반복해서 확인했을 가능성이 크다.

## 3.3 시맨틱 태그 관련 답변

시맨틱 태그 질문에는 비교적 잘 답했다.

답변 요지는 다음과 같았다.

> `div` 태그를 남발해서 개발하는 것보다 의미에 맞는 시맨틱 태그를 사용하는 편이 개발자 입장에서도 구조를 이해하기 쉽고, 개발자 도구로 확인했을 때도 구조가 더 명확하다고 생각했다.

완벽한 답변은 아니었지만 실제 코드에서 사용한 이유를 자신의 언어로 설명했다는 점은 긍정적이다.

더 완성된 답변은 다음과 같다.

> 시맨틱 태그는 각 영역의 의미를 명확하게 표현해 개발자가 구조를 이해하기 쉽게 만들고, 검색엔진과 스크린 리더가 콘텐츠의 역할을 더 잘 해석할 수 있도록 돕습니다. 그래서 `div`만 사용하는 대신 `header`, `nav`, `main`, `section`, `article`, `footer` 등을 역할에 맞게 사용했습니다.

## 3.4 AI 관련 답변

AI 관련 질문에서는 준비했던 내용을 잘 연결해서 답변한 것으로 느껴졌다.

다음과 같은 관점이 잘 전달되었을 가능성이 있다.

- AI를 단순히 코드를 대신 작성해주는 도구로 보지 않음
- AI가 제시한 결과를 그대로 사용하는 것이 아니라 검증하고 판단함
- 반복 코드, 구조 초안, 에러 분석 등에 활용함
- 최종 결정은 개발자가 내린다고 생각함
- AI 시대에는 구현 능력뿐 아니라 맥락 판단, 사용자 흐름 설계, 결과 평가 능력이 중요하다고 봄

AI 관련 답변은 기술 개념 질문보다 본인의 생각과 경험을 연결해 말할 수 있었기 때문에 강점이 더 잘 드러난 부분이었다.

## 3.5 인성 및 태도 관련 답변

인성 질문에서는 항상 배우려는 태도와 겸손함이 중요하다고 답했다.

단순히 말로만 “배우겠다”고 한 것이 아니라, 실제로 학교에서 배우지 않은 영역을 혼자 공부해서 포트폴리오와 프로젝트를 완성했다는 경험이 있기 때문에 어느 정도 행동으로도 증명된 답변이었다.

면접관이 다음과 같이 볼 수 있는 부분이다.

- 모르는 것을 억지로 아는 척하지 않음
- 자신의 부족한 부분을 인정함
- 새로운 내용을 배우려는 태도가 있음
- 독학을 통해 실제 결과물을 만들어본 경험이 있음
- 피드백을 받아들이고 성장할 가능성이 있음

## 4.2 BEM 네이밍 질문

면접관이 개발자 도구에서 클래스명을 보면서 `__`를 왜 사용했는지, `--`와는 무엇이 다른지 물었다.

당시에는 제대로 답변하지 못했다.

이는 BEM 네이밍 규칙에 대한 질문이었다.

BEM은 다음 구조를 사용한다.

```text
Block__Element--Modifier
```

예시:

```scss
.about-hero {
}
.about-hero__character {
}
.about-hero__character--active {
}
```

### `__`: Element

블록 내부의 구성 요소를 의미한다.

```scss
.about-hero__title
.about-hero__character
.about-hero__description
```

### `--`: Modifier

블록이나 요소의 상태 또는 변형을 의미한다.

```scss
.about-hero--dark
.about-hero__character--active
.about-hero__button--disabled
```

면접 답변 예시:

> BEM 네이밍 방식을 사용했습니다. `__`는 블록 안의 특정 요소를 구분할 때 사용하고, `--`는 같은 블록이나 요소의 상태 또는 스타일 변형을 표현할 때 사용합니다. 예를 들어 `about-hero__character`는 `about-hero` 블록 안의 캐릭터 요소이고, `about-hero__character--active`는 해당 요소가 활성화된 상태를 의미합니다.

이 질문을 답하지 못한 것은 아쉬운 부분이다.

특히 본인의 코드에 실제로 적용된 네이밍 규칙이었기 때문에 면접관은 다음을 의심했을 수 있다.

- 규칙을 이해하고 사용한 것인지
- 관습적으로 따라 쓴 것인지
- AI가 작성한 코드를 그대로 사용한 것인지
- 본인이 직접 작성한 코드인지
- 기술 선택을 설명할 수 있는지

다만 이 질문 하나만으로 면접 전체가 부정적으로 결정되었다고 보기는 어렵다.

## 4.3 SEO 질문

면접관은 SEO가 무엇인지, 실제로 어떻게 사용되는지 물었다.

당시에는 다음과 같이 답했다.

> 검색엔진을 통해 웹의 접근성을 높이는 데 사용된다.

방향은 완전히 벗어나지는 않았지만, SEO와 웹 접근성의 개념이 섞인 답변이었다.

정확한 정의는 다음과 같다.

> SEO는 Search Engine Optimization의 약자로, 검색엔진이 웹페이지의 내용을 잘 수집하고 이해해서 검색 결과에 적절히 노출할 수 있도록 개선하는 작업이다.

프론트엔드에서 적용할 수 있는 SEO 요소는 다음과 같다.

- 시맨틱 마크업
- 적절한 `<title>`
- `meta description`
- 올바른 heading 구조
- 이미지 `alt`
- 페이지 성능 개선
- 모바일 대응
- 서버 렌더링 또는 정적 생성
- sitemap
- robots.txt
- Open Graph 메타데이터

Next.js 기준 답변 예시:

> SEO는 검색엔진이 페이지 내용을 잘 수집하고 이해해서 검색 결과에 적절히 노출하도록 최적화하는 작업입니다. 프론트엔드에서는 시맨틱 태그, 페이지별 메타데이터, 제목 구조, 이미지 alt, 성능 개선 등을 적용할 수 있습니다. Next.js에서는 metadata 설정, 서버 렌더링 또는 정적 생성, sitemap과 robots.txt 등을 활용할 수 있습니다.

## 4.4 모르는 질문에 대한 답변 방식

초반에는 모르는 질문을 받으면 다음과 같이 답했다.

> 잘 모르겠습니다. 추후에 공부해보도록 하겠습니다.

그러나 면접관의 표정이나 반응이 다소 짧은 답변을 원하는 것처럼 느껴졌고, 뒤로 갈수록 간단하게 “잘 모르겠습니다”라고 답하게 되었다.

이는 잘못된 대응은 아니다.

오히려 빠르게 여러 개념을 확인하는 면접에서는 매번 긴 마무리를 붙이는 것보다 간결하게 답하는 편이 낫다.

앞으로는 다음과 같은 방식이 적절하다.

### 완전히 모를 때

> 정확히는 모르겠습니다.

### 일부만 알 때

> 정확한 정의는 모르지만, 제가 이해한 범위에서는 ○○와 관련된 개념으로 알고 있습니다.

### 직접 다뤄보지 않았을 때

> 해당 영역은 직접 깊게 다뤄보지 않아 정확하게 답변드리기 어렵습니다.

### 억지로 답하지 않아야 하는 이유

- 잘못된 내용을 확신 있게 말하는 것보다 낫다.
- 자신의 지식 범위를 정확히 아는 태도를 보여줄 수 있다.
- 신뢰를 잃지 않는다.
- 이후 학습 방향을 명확히 할 수 있다.

## 5. 면접 중 심리 상태

전공 질문이 계속 이어지고, 답하지 못하는 질문이 생기면서 중간부터 스트레스를 크게 받았다.

자존감이 낮아지고 시선이 아래로 향하는 순간도 있었다.

이 부분은 면접 태도 측면에서 아쉬울 수 있다.

그러나 다음과 같은 심각한 문제는 없었다.

- 무례하게 대응하지 않음
- 변명하지 않음
- 모르는 내용을 아는 척하지 않음
- 면접을 포기하지 않음
- 준비한 질문에서는 다시 답변함
- 프로젝트와 인성 질문은 계속 대응함

따라서 중간에 위축된 모습이 있었다고 해서 면접 전체가 무너진 것은 아니다.

면접관 입장에서는 긴장하거나 압박을 받는 신입 지원자의 모습으로 볼 수 있다.

다음 면접에서는 답을 못한 직후에도 자세를 유지하는 연습이 필요하다.

### 다음 면접에서 유지할 태도

- 답을 못해도 시선을 바로 내리지 않기
- 모른다고 답한 뒤 표정 유지하기
- 다음 질문은 새로운 평가라고 생각하기
- 한 질문을 못한 것을 전체 면접 실패로 연결하지 않기
- 모르는 질문이 나온 뒤에도 목소리 크기 유지하기

## 6. 마지막 채용 조건 관련 대화

면접 마지막에는 다음과 같은 구체적인 이야기가 나왔다.

- 아직 미성년자라는 점
- 10월부터 출근이 가능하다는 점
- 10월 이전에 일을 시작할 경우 별도 논의가 필요하다는 점
- 미성년자 근무와 관련한 행정 처리
- 4대보험 적용 문제
- 관련 내용을 내부적으로 정리해 메일로 보낼 예정
- 해당 내용을 확인한 뒤 최종 취업 여부를 결정할 예정

이 대화는 비교적 긍정적인 신호로 볼 수 있다.

단순히 형식적으로 결과 안내만 하는 것이 아니라, 실제 채용했을 때 발생하는 근무 조건과 행정 문제를 검토하고 있기 때문이다.

면접관이 이미 완전히 부정적으로 판단했다면 미성년자 고용, 조기 근무, 4대보험 처리 방식까지 구체적으로 이야기할 필요는 적다.

다만 이것만으로 합격이 확정된 것은 아니다.

현재 상태는 다음과 같이 해석하는 것이 가장 적절하다.

> 기술 평가와 프로젝트 검토는 진행되었고, 채용 가능성은 열려 있다. 다만 미성년자 여부, 실제 출근 가능 시점, 4대보험 및 행정 처리 가능 여부를 포함해 최종 판단이 필요한 상태다.

## 7. 면접관이 나를 어떻게 봤을 가능성이 있는가

전체 정황을 종합하면 다음과 같은 평가를 받았을 가능성이 있다.

### 긍정 평가

- 직접 만든 프로젝트의 완성도가 높다.
- 디자인과 개발을 함께 다룰 수 있다.
- 학교에서 배우지 않은 내용을 독학으로 습득했다.
- 결과물을 실제로 끝까지 완성한 경험이 있다.
- 사용자 경험과 화면 디테일에 관심이 있다.
- AI를 도구로 활용하는 기준이 있다.
- 배우려는 태도가 있다.
- 모르는 내용을 억지로 아는 척하지 않는다.
- 프로젝트를 자신의 경험과 연결해서 설명할 수 있다.

### 부정 평가

- 웹 기본 개념이 체계적으로 정리되어 있지 않다.
- 일부 용어를 정확하게 정의하지 못한다.
- 자신의 코드에 적용된 규칙을 설명하지 못하는 부분이 있다.
- 포트폴리오 완성도에 비해 전공 기본기가 들쭉날쭉하다.
- 압박을 받으면 자신감과 시선이 무너질 수 있다.
- 바로 실무에 투입하기에는 추가 학습이 필요할 수 있다.

### 가장 가능성 높은 종합 평가

> 기본 개념은 더 보완해야 하지만, 직접 결과물을 만들 줄 알고 디자인 감각과 독학 능력이 있으며 성장 가능성이 있는 신입 지원자.

## 9. 이번 면접에서 확인된 강점

## 9.1 결과물을 완성하는 힘

단순히 공부만 한 것이 아니라 실제 웹 포트폴리오와 프로젝트를 완성했다.

개인 포트폴리오는 단순 프로젝트 나열이 아니라 스토리와 인터랙션을 설계했다.

효잇과 NOVA는 개발뿐 아니라 디자인에도 직접 참여했다.

이는 신입 지원자에게 중요한 강점이다.

## 9.2 디자인과 프론트엔드의 결합

코드를 작성하는 것뿐 아니라 화면의 정보 구조, 사용 흐름, 시각적 디테일을 함께 생각한다.

피코소프트처럼 고객사 웹사이트나 실제 화면 결과물을 만드는 회사에서는 이러한 역량이 실무적인 장점이 될 수 있다.

## 9.3 독학 능력

학교에서 프론트엔드와 디자인을 체계적으로 배우지 않았지만, 필요한 내용을 스스로 찾아 적용했다.

이는 현재의 지식 공백보다 앞으로의 성장 가능성을 보여주는 요소다.

## 9.4 자신의 경험을 말로 연결하는 능력

준비한 프로젝트와 인성 질문에서는 단순 암기보다 자신의 실제 경험을 연결해서 답했다.

이 부분은 개념 정의형 질문보다 강하게 나타났다.

## 9.5 정직한 태도

모르는 내용을 아는 척하지 않았다.

신입 면접에서는 완벽한 지식보다 신뢰할 수 있는 태도도 중요하다.

## 10.1 구현 중심으로 공부하고 개념 정리가 부족함

- 시맨틱 태그는 사용했지만 검색엔진, 접근성까지 완전히 정리되지는 않음

4. 내 프로젝트에서 어디에 적용했는가
5. 사용하면서 어떤 문제가 있었는가

## 11. 다음 면접을 위한 학습 방식

개념을 무작정 길게 공부하기보다 각 주제를 네 줄로 정리한다.

### 기본 정리 방식

1. 한 문장 정의
2. 왜 필요한가
3. 어디에 적용하는가
4. 내 프로젝트 사례

### SEO 예시

**정의**

검색엔진이 웹페이지를 잘 수집하고 이해해 검색 결과에 적절히 노출하도록 최적화하는 작업이다.

**필요성**

검색을 통해 사용자가 서비스에 유입될 수 있도록 하고, 페이지 내용을 검색엔진에 정확히 전달하기 위해 필요하다.

**적용 방법**

시맨틱 태그, metadata, heading 구조, alt 속성, 성능 개선, sitemap, robots.txt 등을 적용한다.

**프로젝트 사례**

개인 포트폴리오에서 Next.js metadata와 시맨틱 태그를 사용했다.

### BEM 예시

**정의**

CSS 클래스명을 Block, Element, Modifier 구조로 작성하는 네이밍 방법이다.

**필요성**

컴포넌트 구조와 요소 관계를 클래스명만으로 이해하기 쉽고, 스타일 충돌을 줄일 수 있다.

**적용 방법**

`block__element--modifier` 형식을 사용한다.

**프로젝트 사례**

개인 포트폴리오에서 `about-hero__character`와 같은 클래스명을 사용했다.

## 14. 최종 평가

이번 면접은 편안하게 준비한 답변만 말하는 면접이 아니었다.

실제 포트폴리오를 코드 수준으로 확인했고, 여러 프로젝트를 깊게 검증했으며, 웹 개발 전반의 기본 개념을 폭넓게 질문했다.

그 과정에서 다음 약점이 드러났다.

- 전공 개념을 정확히 정의하는 능력 부족
- 포트폴리오에 사용한 일부 기술과 규칙에 대한 설명 부족
- 압박 상황에서 자신감이 낮아지는 모습

그러나 동시에 다음 강점도 분명히 드러났다.

- 직접 완성한 프로젝트의 높은 완성도
- 디자인과 개발을 함께 수행한 경험
- 학교 교육 없이 독학으로 성장한 과정
- 시맨틱 태그 등 실제 코드 품질에 대한 관심
- AI를 활용하는 기준과 태도
- 배우려는 자세와 겸손함
- 실제 채용 조건을 논의할 정도의 회사 측 관심

따라서 이번 면접은 “전공 질문을 몇 개 못했기 때문에 망한 면접”으로 정리하면 안 된다.

더 정확한 평가는 다음과 같다.

> 구현 경험과 결과물, 디자인 감각, 독학 능력은 강하게 보여줬지만, 개념을 정확하게 정의하고 자신의 코드 선택을 설명하는 기본기에서는 보완점이 드러난 면접이었다.

이번 면접의 가장 큰 수확은 다음 면접에서 무엇을 준비해야 하는지가 명확해졌다는 점이다.

포트폴리오를 더 새롭게 만드는 것보다, 이미 만든 결과물에 사용한 기술과 개념을 정확한 언어로 설명하는 훈련이 우선이다.

## 15. 지금의 결론

- 면접은 완전히 부정적이지 않았다.
- 전공 기본기에서는 감점이 있었을 가능성이 높다.
- 프로젝트와 디자인에서는 분명한 가점이 있었다.
- 독학 능력과 성장 가능성은 긍정적으로 전달되었을 가능성이 크다.
- 마지막 근무 조건 논의는 실제 채용 검토가 이어지고 있다는 신호다.
- 결과는 기다려봐야 하지만, 충분히 기대해볼 수 있는 면접이다.
- 합격 여부와 별개로 이번 면접은 앞으로의 준비 방향을 분명하게 만들어준 중요한 경험이다.

## 16. 면접 직후 스스로에게 남기는 말

전공 질문 몇 개를 대답하지 못한 것은 사실이다.

하지만 그 사실이 지금까지 만든 프로젝트와 독학 과정, 디자인 능력, 문제 해결 경험을 없애지는 않는다.

이번 면접에서 부족한 부분이 드러난 것은 실력이 없다는 증거가 아니라, 실전에서 설명해야 하는 범위가 어디까지인지 처음으로 구체적으로 확인한 것이다.

이미 결과물을 만드는 능력은 있다.

이제 필요한 것은 그 결과물에 사용한 개념을 정확한 언어로 정리하는 일이다.

다음 면접에서는 같은 질문을 받았을 때 더 잘 답할 수 있다.
