# 개발자 이력서 작성 실전 가이드

## 1. 이력서의 목적

이력서는 단순히 내가 해온 경험을 나열하는 문서가 아니다.
채용 담당자가 짧은 시간 안에 다음과 같이 판단하도록 만드는 설득 문서이다.

1. 이 지원자는 어떤 개발자인가?
2. 다른 지원자와 무엇이 다른가?
3. 실제로 성과를 낸 경험이 있는가?
4. 면접에서 더 자세히 들어볼 가치가 있는가?

따라서 이력서는 정보의 양보다 **무엇을 먼저 보여주고, 어떤 순서로 설득할 것인지**가 중요하다.

## 2. 피해야 할 구성

### 2.1 노션형 이력서에만 의존하기

노션형 이력서는 깔끔하지만 텍스트가 많아지기 쉽고, 다른 지원자와 비슷해 보일 수 있다.
특히 긴 문장와 여러 개의 토글, 링크가 섞이면 채용 담당자가 핵심 내용을 빠르게 파악하기 어렵다.
따라서 노션은 상세 포트폴리오나 추가 자료를 활용하고, 제출용 이력서는 PDF처럼 레이아웃이 고정된 형태로 만드는 것이 좋다.

### 2.2 첫 페이지에 기술 스택만 나열하기

React, TypeScript, Next.js 와 같은 기술 이름만으로는 실제 역량을 판단하기 어렵다.
기술 스택은 해당 기술을 사용해 무엇을 만들었고, 어떤 문제를 해결했는지와 함께 제시해야 한다.

**좋지 않은 예시**

- React
- TypeScript
- Next.js
- TanStack Query
- Zustand

**개선 예시**

TanStack Query를 활용해 서버 상태를 관리하고, QueryKey를 중앙화하여 데이터 갱신 기준을 통일하였습니다.
기술 자체보다 **기술을 선택한 이유와 활용 결과**가 중요하다.

### 2.3 자기소개를 길게 작성하기

첫 페이지에 긴 자기소개를 넣으면 핵심 강점과 프로젝트가 뒤로 밀릴 수 있다.
자기소개는 한두 문장으로 압축하고, 나머지는 프로젝트 경험으로 증명하는 것이 좋다.

**좋지 않은 예시**
저는 항상 새로운 기술을 배우는 것을 좋아하고,
사용자 경험을 중요하게 생각하며 꾸준히 성장하는 개발자입니다.

**개선 예시**
사용자의 이해 흐름과 상태 변화를 기준으로 화면의 디테일을 설계하는 프론트엔드 개발자입니다.

## 3. 첫 페이지의 핵심 구성

첫 페이지는 채용 담당자가 이력서를 계속 읽을지 결정하는 영역이다.
다음 세 가지가 빠르게 보여야하낟.

### 3.1 프로필

이름, 지원 직무, 연락처, 이메일, GitHub, 포트폴리오 링크 등 기본 정보를 배치한다.
프로필 사진은 필수가 아니지만, 사용할 경우 전체 이력서의 분위기와 맞는 단정한 이미지를 사용한다.

### 3.2 개발자 포지셔닝 한 줄

단순히 직무명만 쓰기보다, 내가 **어떤 기준으로 문제를 해결하는 개발자**인지 표현한다.

**일반적인 표현**

프론트엔드 개발자 김봄입니다.

**포지셔닝이 포함된 표현**

생각의 깊이가 디테일을 만든다. 사용자의 이해 흐름을 설계하는 프론트엔드 개발자 김봄입니다.

포지셔닝 문장은 다음과 같은 질문에 답해야 한다.

> 나는 어떤 문제를 잘 해결하는 개발자인가?

### 3.3 핵심 강점 3가지

강점은 추상적인 성격보다 경험과 성과를 중심으로 작성한다.

**좋지 않은 예시**

책임감이 강합니다.
꾸준히 공부합니다.
협업을 잘 합니다.

**개선 예시**

스크롤 스토리텔링 기반 포트폴리오를 직접 기획·디자인·개발

관리자 웹의 API·Query·Mapper 구조를 설계하고 협업 기준 문서화

SW 마이스터고 연합 해커톤 장관상 1위 및 AppJam 우수상 수상

강점은 가능한 하나의 구체적인 프로젝트, 역할, 결과와 연결한다.

## 4. 프로젝트 작성 전략

프로젝트는 기능 목록이 아니라 다음 구조로 작성한다.

### 4.1 프로젝트 개요

프로젝트가 어떤 문제를 해결하는 서비스인지 한두 문장으로 설명한다.

세탁기와 예약, 고장 신고를 관리하는 관리자 웹 서비스입니다.
관리자가 여러 데이터를 빠르게 확인하고 처리할 수 있도록 설계했습니다.

### 4.2 담당 역할

팀 전체가 한 일을 쓰지 말고, 내가 직접 맡은 영역을 명확하게 구분한다.

프론트엔드 개발자 2명 중 한 명으로 참여했습니다.

API 요청 구조 설계
TanStack Query 기반 서버 상태 관리
예약 및 기기 관리 화면 구현
서버 응답 Mapper 기준 정리
협업 문서 작성

### 4.3 문제 상황

어떤 문제나 제약이 있었는지 설명한다.

API마다 응답 구조가 달라 화면에서 데이터를 직접 가공하는 코드가 반복됐습니다.
또한 팀원마다 서버 응답을 처리하는 기준이 달라 코드의 일관성이 떨어졌습니다.

### 4.4 해결 과정

무엇을 고민했고, 왜 해당 방법을 선택했는지를 작성한다.

서버 응답을 무조건 반환하기보다,
화면에서 직접 사용하기 어려운 데이터에만 Mapper을 적용하는 기준을 세웠다.

API, URL, HTTP Client, API 함수, Query Hook, UI 순서로 요청 구조를 분리하고, QueryKey를 중앙화하여 데이터 갱신 기준을 통일했습니다.

### 4.5 결과

가능하면 숫자나 명확한 변화로 표현한다.
숫자를 억지로 만들 필요는 없지만, 측정할 수 있는 결과가 있다면 적극적으로 사용한다.

예약 삭제 후 관련 목록을 한 번에 갱신할 수 있도록 QueryKey 구조를 개선했습니다.

팀원이 새로운 API를 추가할 때 동일한 구조를 참고할 수 있도록 문서화했습니다.

화면 컴포넌트에서 직접 데이터를 가공하는 중복 코드를 줄였습니다.

## 5. 숫자를 활용하는 방법

숫자는 강한 인상을 주지만, 근거 없이 과장해서 사용하면 안 된다.

### 사용할 수 있는 숫자

- 프로젝트 기간
- 팀 인원
- 담당 화면 수
- 구현 기능 수
- 커밋 수
- 사용자 수
- 성능 개선 수치
- 오류 감소율
- 응답 속도 개선
- 수상 순위
- 참여 인원
- 운영 기간

### 예시

약 10개월 동안 개인 포트폴리오를 제작하며 400개 이상의 커밋을 기록했습니다.
5개의 전공 캐릭터를 직접 제작하고 React Native 화면에 적용했습니다.
SW 마이스터고 연합 해커톤에서 1위를 수상했습니다.
측정하지 않은 성능을 임의로 숫자로 표현해서는 안된다.

> 사용성을 80% 개선했습니다.
> 성능을 2배 높였습니다.

이런 문장은 근거가 명확할 대만 사용한다.

## 6. 전체 이력서의 4단계 흐름

### 1단계: 호기심 자극

첫 페이지에서 지원자의 포지셔닝과 핵심 강점을 보여준다.

채용 담당자가 다음과 같이 느껴지도록 만드는 단계이다.

> 이 지원자는 다른 사람과 어떤 점이 다르지?

**포함할 내용**

- 이름과 지원 직무
- 포지셔닝 한 줄
- 핵심 강점 3가지
- 대표 성과
- 연락처와 포트폴리오 링크

### 2단계: 호기심 증폭

대표 프로젝트를 통해 첫 페이지에서 주장한 강점을 보여준다.

**포함할 내용**

- 대표 프로젝트
- 프로젝트 목적
- 본인의 역할
- 핵심 기술
- 눈에 띄는 결과
- 이미지나 화면 자료

단순 기능 나열보다 문제와 해결 과정이 보여야 한다.

### 3단계: 불안 해소와 신뢰 형성

지원자가 실제로 해당 경험을 수행했는지 근거를 제공한다.

**포함할 내용**

- 기술 선택 이유
- 문제 해결 과정
- 협업 방식
- 설계 기준
- 트러블 슈팅
- 수상 및 교육 경험
- Github, 배포 링크, 포트폴리오 링크

이 단계에서는 화려한 표현보다 구체적인 설명이 중요하다.

### 4단계: 행동 유도

채용 담당자가 포트폴리오를 확인하거나 면접을 제안하도록 연결한다.

**예시**
프로젝트의 상세 설계와 구현 과정은 포트폴리오에서 확인할 수 있습니다.

**Portfolio**
https://example.com

**Email**
example@email.com

과한 문구를 넣기보다 링크와 연락처를 명확하게 제공하면 된다.

## 7. 시각적 구성 전략

이력서는 디자인 작품이 아니라 정보를 빠르게 전달하는 문서이다.

### 좋은 시각적 구성

- 제목과 본문의 크기 차이가 명확함
- 한 페이지에 핵심 메시지가 하나씩 있음
- 문장보다 짧은 항목 중심
- 중요 결과가 먼저 보임
- 여백이 충분함
- 프로젝트별 구조가 통일됨
- 링크를 쉽게 구분할 수 있음
- PDF로 열었을 때 레이아웃이 깨지지 않음

### 피해야 할 구성

- 지나치게 작은 글씨
- 기술 로고의 과도한 사용
- 장식만을 위한 그래픽
- 배경과 글자의 낮은 대비
- 너무 많은 색상
- 근거 없는 그래프와 숫자
- 사진이 콘텐츠보다 큰 구성
- 프로젝트 설명이 이미지에만 포함된 형태

## 8. 이미지 기반 이력서 활용 시 주의점

시각적으로 구성된 PDF 이력서는 지우너자를 기억하게 만드는 데 도움이 될 수 있다.
하지만 모든 기업에서 이미지 중심 이력서가 유리한 것은 아니다.

특히 다음 요소를 함께 고려해야한다.

- 텍스트 검색과 복사가 가능한 PDF인지
- 모바일과 데스트톱 모두 읽기 쉬운지
- 채용 시스템에서 파일이 정상적으로 열리는지
- 디자인보다 프로젝트 내용이 먼저 보이는지
- 기술 역량에 대한 근거가 충분한지

따라서 이력서는 포스터처럼 만들기보다, **읽기 쉬운 문서에 적절한 시각적 강조를 추가하는 방식** 이 안전하다.

## 10. 핵심 요약

합격 가능성을 높이는 이력서는 많은 내용을 담은 이력서가 아니다.
채용 담당자가 짧은 시간 안에 다음 내용을 이해할 수 있는 이력서이다.

> 이 지원자는 어떤 개발자인가?
> 무엇을 잘하는가?
> 그 강점을 어떤 경험으로 증명했는가?
> 왜 면접에서 더 자세히 들여봐야 하는가?

핵심 원칙은 다음과 같다.

1. 첫 페이지에서 개발자 포지셔닝을 명확히 한다.
2. 추상적인 소개보다 구체적인 강점과 성과를 먼저 보여준다.
3. 기술 이름보다 해당 기술로 해결한 문제를 작성한다.
4. 프로젝트는 문제, 역할, 해결, 결과 순서로 구성한다 .
5. 숫자는 실제 근거가 있을 때만 사용한다.
6. 시각적 디자인은 정보 전달을 돕는 수준으로 사용한다.
7. 이력서 전체를 면접으로 연결되는 설득 흐름으로 설계한다.
