# 프론트엔드 역량 진단 및 보완 계획

> 현재 상태를 감정적으로 평가하기보다, 실제 면접에서 드러난 약점과 강점을 기준으로 앞으로의 학습 방향을 정리한 문서  
> 핵심 목표: **이미 만든 결과물을 웹 기본기와 연결해 정확하게 설명할 수 있는 개발자 되기**

---

# 1. 현재 상태 요약

현재 가장 큰 문제는 개발을 못한다는 것이 아니다.

이미 다음과 같은 기술과 프로젝트 경험을 가지고 있다.

- Next.js
- React
- TypeScript
- GSAP
- ScrollTrigger
- Lenis
- SCSS
- TanStack Query
- Zustand
- 반응형 웹
- 시맨틱 마크업
- API 구조 설계
- 상태 관리
- 컴포넌트 설계
- 프로젝트 디자인
- 사용자 흐름 설계
- AI 활용
- 실제 오류 해결

문제는 이 경험이 **정확한 개념과 기술 언어로 정리되지 않았다는 점**이다.

현재 상태를 한 문장으로 정리하면 다음과 같다.

> 결과물을 만들 수 있고 감각과 실행력은 강하지만, 지식을 체계적으로 정리하고 기술 선택을 정확한 언어로 설명하는 능력은 아직 부족한 신입 개발자다.

따라서 앞으로는 새 프로젝트를 계속 추가하는 것보다, 이미 만든 프로젝트와 웹 기본기를 연결해서 정리하는 단계로 넘어가야 한다.

# 2. 강점 분석

## 2.1 결과물을 실제로 완성하는 능력

단순히 강의나 예제를 따라 한 것이 아니라, 여러 프로젝트를 직접 기획하고 구현했다.

특히 개인 포트폴리오는 단순 프로젝트 나열형이 아니라 스크롤 흐름, 인터랙션, 정보 구조, 시각적 연출까지 포함해 하나의 경험으로 설계했다.

이 능력은 지식만 알고 있는 지원자와 구분되는 분명한 강점이다.

## 2.2 디자인과 개발을 함께 볼 수 있음

효잇과 NOVA에서는 개발뿐 아니라 디자인에도 직접 참여했다.

다음과 같은 강점이 있다.

- 화면 구조를 생각하는 능력
- 사용자 흐름을 설계하는 능력
- 시각적 디테일을 구현하는 능력
- 인터랙션을 기능과 연결하는 능력
- 사용자가 정보를 이해하는 순서를 고려하는 능력

프론트엔드 개발자에게 이 감각은 분명한 경쟁력이다.

## 2.3 독학 능력

학교 전공이 프론트엔드 중심이 아니었음에도 불구하고 HTML, CSS, JavaScript, React, TypeScript, 디자인, 인터랙션 등을 대부분 독학으로 익혔다.

이는 다음 역량을 보여준다.

- 필요한 것을 스스로 찾는 능력
- 교육 과정 밖의 내용을 확장하는 능력
- 실제 프로젝트에 적용하는 능력
- 끝까지 완성하는 능력

## 2.4 프로젝트 기반 사고

기술을 단순히 사용한 것이 아니라 프로젝트에서 문제를 해결하기 위해 적용했다.

예:

- Query Key 중앙화
- Mapper 적용
- Raw Response 기준 논의
- 상태 기반 UI
- 부모와 자녀 역할 분리
- 인터랙션 흐름 설계
- 반응형 구조
- API 계층 분리

이런 경험은 면접에서 제대로 언어화할 수만 있다면 강한 경쟁력이 된다.

## 2.5 AI 활용 기준이 있음

AI를 그대로 의존하는 것이 아니라 다음과 같은 방식으로 활용하려는 기준이 있다.

- 반복 코드 보조
- 구조 초안 작성
- 오류 분석
- 코드 검토
- 대안 비교
- 최종 판단은 직접 수행

AI 시대에 중요한 것은 도구 사용 여부보다 결과를 평가하고 수정하는 능력이다.

# 3. 약점 분석

## 3.1 구현 경험이 개념으로 정리되지 않음

현재 가장 큰 약점이다.

실제로 사용한 기술은 많지만, 면접에서 정확한 정의나 차이를 바로 설명하지 못하는 경우가 있다.

예:

- BEM을 사용했지만 `__`, `--` 차이를 설명하지 못함
- SEO를 알고 있지만 접근성과 섞어서 설명함
- 시맨틱 태그를 사용했지만 구조 가독성 중심으로만 설명함
- 프로젝트 구조를 만들었지만 이론적 기준을 한 문장으로 압축하기 어려움
- 상태 관리와 서버 상태를 구분해 설명하기 어려운 순간이 있음
- 렌더링과 화면 업데이트를 혼합해서 설명할 가능성이 있음

즉, 지금은 다음 상태에 가깝다.

> 할 줄은 알지만, 왜 그렇게 했는지 정확히 설명하는 능력이 부족하다.

이 문제를 해결하려면 모든 기술을 다음 다섯 가지로 정리해야 한다.

1. 무엇인가
2. 왜 필요한가
3. 왜 이 기술을 선택했는가
4. 어디에 적용했는가
5. 장점과 한계는 무엇인가

---

## 3.2 웹 기본 개념의 빈칸이 들쭉날쭉함

프로젝트에서 필요한 내용을 그때그때 학습했기 때문에, 깊게 아는 영역과 거의 모르는 영역의 차이가 크다.

### 비교적 강한 영역

- 화면 구조
- 사용자 흐름
- 디자인
- 인터랙션
- 프로젝트 설계
- AI 활용
- 실제 문제 해결
- 컴포넌트 분리
- 상태 기반 UI

### 약한 영역

- HTML/CSS 용어 정의
- CSS 네이밍 방법론
- 시맨틱 마크업의 전체 목적
- SEO
- 웹 접근성
- HTTP
- 세션, 쿠키, JWT
- 브라우저 동작
- JavaScript 핵심 개념
- 네트워크 기본
- 웹 전반의 공통 용어
- MVC, MVP, MVVM 같은 소프트웨어 구조 개념

이는 독학형 개발자에게 흔한 패턴이다.

실제 구현을 먼저 배웠기 때문에 “어떻게 만드는지”는 알지만, “정확한 이름과 개념의 경계”가 비어 있는 상태다.

---

## 3.3 자기 코드에 대한 검증 대비가 부족함

포트폴리오를 만들 때 결과물 완성에 집중했지만, 코드 한 줄 단위로 질문받는 상황까지 준비하지 못했다.

예:

```scss
.about-hero__character
```

이 클래스명 하나만 있어도 다음 질문이 나올 수 있다.

- 왜 이렇게 이름을 지었는가?
- BEM인가?
- `__`와 `--`의 차이는 무엇인가?
- 클래스 충돌은 어떻게 방지하는가?
- CSS Module 대신 SCSS를 쓴 이유는?
- 유지보수 측면에서 어떤 장점이 있는가?
- modifier를 실제로 사용한 사례는 있는가?

앞으로는 프로젝트의 기능 설명뿐 아니라, 코드에 남은 모든 선택을 질문 대상으로 봐야 한다.

---

## 3.4 정확하지 않을 때 두루뭉술하게 답하는 경향

아예 모르는 것은 아니지만 확신이 없을 때 비슷한 개념을 섞어서 말하는 경향이 있다.

예:

> SEO는 검색엔진으로 웹의 접근성을 높이는 것입니다.

이 답변은 방향은 일부 맞지만 SEO와 접근성이 혼합되어 있다.

정확한 답변은 다음과 같다.

> SEO는 검색엔진이 웹페이지의 내용을 수집하고 이해해서 검색 결과에 적절히 노출할 수 있도록 최적화하는 작업입니다.

면접에서는 다음 개념이 혼합될 가능성이 있다.

- SEO와 웹 접근성
- 상태와 데이터
- 구조와 아키텍처
- 서버 상태와 클라이언트 상태
- 렌더링과 화면 업데이트
- SSR과 SEO
- 세션과 JWT
- 쿠키와 브라우저 저장소

문제는 생각이 없는 것이 아니라, **정의를 먼저 말하는 습관이 부족한 것**이다.

앞으로 모든 개념 답변은 다음 순서를 사용한다.

```text
정의 → 목적 → 적용 방법 → 내 경험
```

---

## 3.5 압박을 받으면 이후 답변까지 무너질 수 있음

기술 질문 몇 개를 답하지 못한 뒤 다음과 같은 반응이 나타났다.

- 시선이 아래로 내려감
- 자존감이 빠르게 떨어짐
- 면접을 망쳤다고 판단함
- 다음 답변의 자신감도 줄어듦

면접관은 한 질문을 못했다고 전체 평가를 끝내지 않는다.

그러나 스스로 면접 전체를 실패로 판단하면 실제 역량보다 더 약하게 보일 수 있다.

필요한 것은 모든 질문을 맞히는 능력이 아니다.

> 하나를 틀려도 다음 질문에서 다시 정상적으로 답하는 회복력

이 필요하다.

---

## 3.6 강점과 약점의 격차가 큼

현재 일반적인 신입보다 강한 부분이 분명하다.

- 디자인
- 인터랙션
- 결과물 완성도
- 독학
- 사용자 경험
- 프로젝트 집요함
- 자기주도성

반면 기본 개념 질문에서는 불균형이 드러날 수 있다.

면접관은 다음과 같은 의문을 가질 수 있다.

> 결과물은 좋은데 기본기는 왜 이렇게 불균형한가?

포트폴리오 수준이 높기 때문에 오히려 질문도 더 깊어진다.

따라서 일반 지원자보다 자신의 코드와 기술 선택을 더 깊게 설명할 준비가 필요하다.

# 5. 학습 방향

## 5.1 핵심 원칙

앞으로의 공부는 다음 사이클로 진행한다.

```text
개념 학습 → 내 코드에서 찾기 → 코드 검토 → 한 문장 정의 → 소리 내서 설명
```

책을 읽기만 하거나, 면접 답변만 외우는 방식은 피한다.

### 예시: BEM

1. BEM의 정의를 공부한다.
2. 포트폴리오 코드에서 BEM 클래스명을 찾는다.
3. 잘못 사용한 곳이 있는지 확인한다.
4. 왜 사용했는지 정리한다.
5. 다른 네이밍 방식과 비교한다.
6. 30초 안에 소리 내서 설명한다.

---

## 5.2 공부 비중

당분간 다음 비중으로 진행한다.

| 영역                      | 비중 |
| ------------------------- | ---: |
| 포트폴리오 코드·기술 정리 |  50% |
| 웹 기본 CS                |  40% |
| 말하기 및 압박 대응       |  10% |

CS만 공부하면 프로젝트와 분리된 지식만 쌓일 수 있다.

포트폴리오만 정리하면 새로운 개념 질문에서 다시 막힐 수 있다.

따라서 두 영역을 반드시 연결한다.

# 7. 추가로 보완할 웹 기초 영역

CS 책만으로는 프론트엔드 기본기가 충분히 채워지지 않는다.

다음 영역은 별도 학습이 필요하다.

## HTML

- 시맨틱 태그


# 10. 답변 구조

모든 개념 질문은 다음 순서로 답한다.

```text
정의 → 목적 → 적용 방법 → 내 경험
```

## SEO 예시

> SEO는 검색엔진이 웹페이지의 내용을 수집하고 이해해서 검색 결과에 적절히 노출할 수 있도록 최적화하는 작업입니다. 프론트엔드에서는 시맨틱 태그, metadata, heading 구조, 이미지 alt, 성능 개선 등을 적용할 수 있습니다. 제 포트폴리오에서는 Next.js metadata와 시맨틱 마크업을 사용했습니다.

## BEM 예시

> BEM은 CSS 클래스명을 Block, Element, Modifier 구조로 작성하는 네이밍 방식입니다. `__`는 블록 내부 요소를 나타내고, `--`는 상태나 스타일 변형을 나타냅니다. 제 포트폴리오에서는 `about-hero__character`와 같은 클래스명에 적용했습니다.

## 시맨틱 태그 예시

> 시맨틱 태그는 태그 이름 자체가 콘텐츠의 역할과 의미를 나타내는 HTML 요소입니다. 개발자가 구조를 이해하기 쉽고, 검색엔진과 스크린 리더가 페이지를 해석하는 데 도움이 됩니다. 제 포트폴리오에서는 `header`, `main`, `section`, `footer` 등을 역할에 맞게 사용했습니다.

# 17. 최종 결론

현재 가장 큰 약점은 개발 능력 자체가 아니다.

> 실제로 해본 것에 비해 개념 정리와 설명력이 따라오지 못하는 것

이 핵심이다.

따라서 앞으로의 방향은 다음과 같다.

```text
웹 기본기 학습
→ 내 프로젝트 코드에서 확인
→ 잘못 적용한 부분 수정
→ 기술 선택 이유 정리
→ 한 문장 정의 작성
→ 소리 내서 설명
→ 모의 면접으로 압박 대응
```

이 과정을 반복하면 포트폴리오의 완성도와 기술 설명 수준 사이의 차이가 줄어든다.

새 프로젝트를 더 만드는 것보다, 이미 만든 프로젝트를 정확하게 설명할 수 있는 수준으로 끌어올리는 것이 현재 취업 경쟁력을 가장 크게 높이는 방법이다.
